# LifeOS Public Roadmap

Last updated: `2026-04-01`

LifeOS is an AI-native Linux distribution focused on:

- local-first intelligence
- privacy by default
- sovereign personal computing

This roadmap is the public, readable version of where the project stands today and where it is going next.

It is intentionally simpler than the technical strategy docs, but it follows the same source of truth.

LifeOS is built in Mexico and published for a global open-source audience. That origin is part of the project's identity, not a limit on who the project is for.

---

## What is real today

These foundations are already present in the project today:

- Local inference through the LifeOS AI runtime
- Encrypted local memory foundations
- Desktop control plane foundations
- Remote interaction through Telegram
- Voice, vision, and automation groundwork
- Public branding and landing page foundations

Important:

LifeOS is still early. The goal here is not to pretend the system is finished, but to show what is already real and what is still being hardened.

---

## Now

These are the areas we are actively pushing forward right now.

### 1. Stabilize the public beta foundation

We are making the real system more dependable on real hardware:

- install and first-boot reliability
- update and rollback confidence
- runtime consistency
- host validation of features that already exist in repo

### 2. Improve desktop control and the operator loop

LifeOS is not just a local model runner. It is moving toward an operating system where the assistant can act through the desktop with clearer control and safer boundaries.

Current focus:

- stronger system control surfaces
- better operator workflows
- more reliable remote interaction
- clearer paths from user intent to system action

### 3. Grow public documentation and product clarity

We want the project to be easier to understand from the outside:

- clearer public roadmap
- better demos and onboarding
- more public-facing documentation
- tighter alignment between website, repo, and actual runtime state

---

## Next

These are the next major areas after the current stabilization push.

### Meetings and memory intelligence

The meeting pipeline has advanced strongly in repo. The next step is validating it thoroughly in real host use so that:

- detection works reliably
- transcripts and summaries are trustworthy
- retention stays clean and does not leave useless storage behind

### Security by default

LifeOS is also moving toward a stronger default posture out of the box:

- safer host defaults
- better system hardening
- clearer operational guardrails

### Public rails for following and supporting the project

The project now has a public landing page, and the next layer is to connect it to:

- roadmap visibility
- support rails
- demos and public updates

---

## Later

These are meaningful directions for LifeOS, but they are not the immediate critical path.

- richer personalization and adaptive workflows
- stronger mobile and cross-platform control
- deeper knowledge and memory systems
- local training and specialized models
- broader user-facing product surfaces beyond the current core

These remain important, but they should not outrun the real stability of the base system.

---

## Path to public beta

For LifeOS, "public beta" means something specific.

It does **not** mean:

- every long-term vision item is complete
- every experimental subsystem is perfect
- every platform is supported

It **does** mean:

- the core install and update path is stable
- local AI and memory foundations are usable
- desktop control is understandable and reliable
- the project is documented clearly enough for outsiders to try and follow

---

## How we describe progress

We now try to avoid vague claims like "done" unless they are clearly grounded.

When possible, a feature should fit one of these realities:

- validated on a real host
- integrated in repo
- partial / still being wired end-to-end

That honesty matters because LifeOS is being built in public.
