# lifeos-tts container

Phase 1 of the [architecture pivot](../../docs/strategy/prd-architecture-pivot-lean-bootc-quadlet.md). First service to be containerized as a Quadlet.

## Status

**Scaffold — NOT yet active.** This directory contains the Containerfile + Quadlet definition that will replace the host-systemd `lifeos-tts-server.service` once Phase 1 officially flips. The files are committed for review and so the dev workflow described in [`docs/contributor/quadlet-dev-workflow.md`](../../docs/contributor/quadlet-dev-workflow.md) can already build/push to the dev registry.

## What this replaces

Today TTS runs on the host:

- `/etc/systemd/system/lifeos-tts-server.service` (DynamicUser, hardened)
- `/opt/lifeos/kokoro-env/` (Python venv baked in bootc image)
- `/usr/local/bin/lifeos-tts-server.py` (the server script)

After Phase 1 flips:

- `/etc/containers/systemd/lifeos-tts.container` (Quadlet, generated by `lifeos-tts.service`)
- The Python venv lives **inside** the container (no longer in the bootc image — saves ~850 MB on the rootfs)
- `lifeos-tts-server.py` stays on the host, bind-mounted into the container read-only — so script edits don't require a container rebuild

## Why TTS first

- **Pure CPU.** No GPU passthrough, no CDI, no Vulkan. Phase 1 validates the Quadlet pattern WITHOUT also having to debug NVIDIA Container Toolkit (which has a known Fedora 43 RPM digest issue — see `project_pending_nvidia_container_toolkit` memory).
- **Stateless.** Kokoro voice models are baked into the Python venv (`kokoro==0.9.4` package ships them), so the container needs zero bind mounts to `/var/lib/lifeos/`. If the container dies and restarts, no state to recover.
- **Loosely coupled.** Clients (lifeosd, dashboard) reach `:8084` over HTTP. No IPC, no shared FDs, no D-Bus. If the migration breaks TTS, only TTS breaks — Axi keeps responding via dashboard text.

## Build (dev iteration)

```bash
cd ~/dev/gama/lifeos/lifeos
podman build -t 10.66.66.1:5001/lifeos-tts:dev \
  -f containers/lifeos-tts/Containerfile \
  containers/lifeos-tts/
```

Expected size: ~850 MB (torch CPU wheels dominate).

## Push to dev registry

```bash
podman push --tls-verify=false 10.66.66.1:5001/lifeos-tts:dev
```

The dev registry is private over WireGuard (`10.66.66.1:5001`). Tags `:dev`, `:branch-*`, `:sha-*` get auto-pruned after 72h by `lifeos-registry-gc.timer`. See [`docs/contributor/quadlet-dev-workflow.md`](../../docs/contributor/quadlet-dev-workflow.md).

## Test on laptop (manual until Phase 1 flips)

```bash
ssh laptop "
  # 1. Stop the legacy service
  sudo systemctl stop lifeos-tts-server.service
  sudo systemctl mask lifeos-tts-server.service

  # 2. Pull and tag the dev image
  podman pull --tls-verify=false 10.66.66.1:5001/lifeos-tts:dev
  podman tag 10.66.66.1:5001/lifeos-tts:dev localhost/lifeos-tts:current

  # 3. Drop the Quadlet (one-time bootstrap; future Phase 1 build will install it)
  sudo cp /var/home/lifeos/.../containers/lifeos-tts/lifeos-tts.container \
          /etc/containers/systemd/lifeos-tts.container
  # Patch Image= line to point at localhost/lifeos-tts:current for dev iteration
  sudo sed -i 's|^Image=.*|Image=localhost/lifeos-tts:current|' \
          /etc/containers/systemd/lifeos-tts.container
  sudo systemctl daemon-reload
  sudo systemctl start lifeos-tts.service

  # 4. Smoke test
  curl -s http://127.0.0.1:8084/health | jq .
  curl -s http://127.0.0.1:8084/voices | jq '.voices | length'
"
```

## Promote to production

After validation passes:

```bash
podman tag 10.66.66.1:5001/lifeos-tts:dev ghcr.io/hectormr206/lifeos-tts:stable
podman push ghcr.io/hectormr206/lifeos-tts:stable
```

Next bootc image release will include the Quadlet pointing at `:stable`.

## Rollback if anything breaks

```bash
ssh laptop "
  sudo systemctl stop lifeos-tts.service
  sudo systemctl unmask lifeos-tts-server.service
  sudo systemctl start lifeos-tts-server.service
"
```

Restores the host-systemd path. Zero data loss (TTS is stateless).

## Trade-offs accepted in this design

| Decision | Why we picked it | Trade-off accepted |
|---|---|---|
| Bind-mount the `.py` script (not COPY into image) | Iteration on prompt/route logic doesn't need a container rebuild | Script changes don't go through CI image build — slight risk of drift between source and what's running |
| `Network=host` | Zero URL changes in lifeosd; existing clients keep using `127.0.0.1:8084` | Less isolation than a podman bridge — accepted because everything is loopback-only anyway |
| Restart=always, no WatchdogSec yet | The Python script doesn't sd_notify. Adding that is a follow-up; for now Restart=always handles crashes | A hung-but-not-dead container won't restart until next failure event |
| Bake Kokoro venv into image (not bind mount) | Container is self-contained; new contributors can just `podman run` it without any host setup | When Kokoro updates, the container image needs a rebuild (acceptable — that's how every other ML service works) |
